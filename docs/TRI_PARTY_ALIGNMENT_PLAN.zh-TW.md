# 三方協作一頁式方案（遊戲端 / 舊後台端 / 新後台端）

## 目標
- 不破壞舊後台既有使用。
- 新後台快速到達可與遊戲程式對接的實用程度。
- 欄位語意優先尊重遊戲端既有定義。

## 三方角色與決策邊界
- 遊戲端同事（資料語意主責）
  - 定義玩家、房卡、儲值單、狀態流轉的業務語意。
  - 決定欄位名稱、狀態枚舉、必要驗證規則。
- 舊資料庫/舊後台同事（相容與穩定主責）
  - 維持現行系統可用與資料一致。
  - 提供既有欄位來源與歷史相容注意事項。
- 新後台（本專案）
  - 以遊戲端語意為對外 API 合約。
  - 在內部做舊資料庫映射，不把舊欄位設計直接外露。

## 核心原則
- 新舊並行：舊後台不中斷，新後台獨立演進。
- 對外一致：新後台 API 合約以遊戲端語意為準。
- 內部相容：透過映射層連舊資料，不強迫舊系統立即改表。
- 可回退：任何寫入功能都保留切回舊流程的路徑。

## 先做什麼（兩週內可落地）
1. 鎖定 V1 對接範圍
- Auth：`/api/auth/login`
- Players：`/api/players`、`/api/players/:id`、`/api/players/:id/status`
- Recharge：`/api/recharge-orders`、`/api/recharge-orders/:id/confirm`、`/api/recharge-orders/:id/reject`
- RoomCards：`/api/room-cards/balances`、`/api/room-cards/logs`、`/api/room-cards/adjust`

2. 鎖定欄位字典（單一來源）
- 玩家查詢 key：`externalId`、`nickname`
- 房卡異動來源：`recharge`、`open_table`、`admin_adjust`
- 核帳狀態：`pending`、`confirmed`、`rejected`
- 回傳時間一律 ISO 字串，前端顯示時再本地格式化

3. 寫入策略
- 預設 `DATABASE_ACCESS_MODE=readonly`
- 僅在 staging 開啟寫入驗證
- 寫入功能分批開：先核帳、再房卡調整、最後玩家狀態

## 對接定義（給遊戲端）
- 請求/回應契約固定，不因舊資料庫欄位差異而改動對外命名。
- 搜尋語意固定：同一搜尋詞在餘額與 log 必須一致（已完成）。
- 錯誤格式固定：`{ message: string }`，配合 HTTP status 使用。

## 驗收門檻（Ready to Integrate）
- API 一致性
  - 同條件查詢，列表與詳情不矛盾。
  - 同條件搜尋，餘額與 log 結果一致。
- 流程正確性
  - 核准/駁回後，訂單狀態與房卡異動對得上。
  - 玩家停權/復權後，玩家列表狀態立即可見。
- 安全性
  - readonly 模式下，所有寫入按鈕與 API 都被拒絕。

## 切換與回退
- 切換策略：先「讀」後「寫」，先 staging 後 production。
- 回退條件：
  - 狀態流轉錯誤
  - 欄位語意不一致
  - 寫入後對帳失敗
- 回退方式：關閉新後台寫入路徑，維持舊後台寫入。

## 本週可執行清單
- [ ] 三方確認 V1 欄位字典（30 分鐘會議）
- [ ] 三方確認狀態枚舉與錯誤碼
- [ ] staging 完成核帳與房卡寫入 smoke test
- [ ] 提供遊戲端一份固定 API 合約（不再隨意改名）

## 備註
- 舊後台是相容來源，不是新後台契約來源。
- 新後台對外契約以遊戲端可用、可維護為優先。
