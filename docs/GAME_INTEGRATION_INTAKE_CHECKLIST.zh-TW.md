# 遊戲程式對接收斂清單（不破壞舊系統版）

## 你要先對遊戲同事講的前提
- 我們新後台會以遊戲端欄位語意為準。
- 舊後台會繼續運作，不做一次性替換。
- 新功能先在 staging 對接，確認後再逐步開啟寫入。

## 你要問遊戲同事的問題（必問）

### 1. 物件與欄位定義（Source of Truth）
- 玩家（Player）有哪些欄位是「一定要有」？
- 房卡餘額（RoomCardBalance）對遊戲而言最小必要欄位是什麼？
- 房卡異動（RoomCardLog）必填欄位有哪些？
- 儲值單（RechargeOrder）必填欄位有哪些？
- 哪些欄位只讀、哪些欄位可由後台改？

### 2. 狀態枚舉與流轉規則
- 玩家狀態有哪些合法值？例如 active / banned。
- 儲值單狀態有哪些合法值？例如 pending / confirmed / rejected。
- 每個狀態可以從哪個狀態轉過來？
- 哪些狀態轉換需要備註（reviewNote）？

### 3. 事件順序與一致性要求
- 核准儲值後，遊戲端預期多久內看到房卡增加？
- 若核准成功但房卡沒加成功，遊戲端希望怎麼處理？
- 玩家停權後，遊戲端要立即擋登入，還是下次檢查才生效？

### 4. 查詢語意（避免前後端對不上）
- 玩家查詢關鍵字應匹配哪些欄位？（externalId / nickname / playerId）
- 房卡 log 查詢是否也要用同一套關鍵字語意？
- 時間範圍查詢要用哪個時區與格式？

### 5. 錯誤格式與重試策略
- 遊戲端接 API 失敗時，需要固定哪些錯誤碼？
- 哪些錯誤可以自動重試，哪些不可以？
- 後台回傳 `message` 外，是否還需要 `code` 欄位？

### 6. 安全與權限
- 遊戲端是否需要區分 support / manager / owner 的可見欄位？
- 哪些 API 要強制紀錄 audit log？
- 需要哪些防重放/防重複提交機制？

## 你要跟遊戲同事「索取」的資料（必拿）
- 資料字典（欄位名稱、型別、可空、範例值）。
- 狀態機文件（狀態值 + 流轉圖 + 非法轉換）。
- API 契約草稿（request/response JSON 範例）。
- 錯誤碼表（HTTP status + 業務錯誤碼 + UI 呈現文案）。
- 3~5 筆真實樣本資料（去識別化即可）。

## 對接會議 30 分鐘建議流程
1. 先定義「不變動的欄位語意」
2. 再定義「可晚點做的欄位/功能」
3. 確認狀態流轉與錯誤碼
4. 決定第一批對接 API 範圍

## 第一批對接 API（建議）
- Auth：登入
- Players：列表、單筆、狀態切換
- RechargeOrders：列表、核准、駁回
- RoomCards：餘額、異動紀錄

## 不破壞舊運作的施工規則
- 新路徑先讀後寫：先完成查詢一致，再開寫入。
- 先 staging 再 production：每批 API 都要先驗收。
- 寫入加 feature flag：可隨時關閉回舊流程。
- 任何欄位改名先做映射，不直接改舊資料欄位。
