# 後台能力藍圖（玩家需求優先）

## 目的
- 先以玩家會提出的客服需求為核心，反推後台能力。
- 再用市面同類型遊戲常見服務做補齊，避免漏掉關鍵功能。
- 最後整理成客服方便看、方便操作、方便追溯的資訊架構。

## 一、玩家事件驅動能力地圖（客服實務版）
1. 玩家說我有儲值但沒入帳
- 客服要先查：訂單狀態、金額、時間、玩家帳號
- 客服要能做：建立待審單、補充證據、交給主管審核
- 後台必要功能：儲值單查詢、待審佇列、核准/駁回、稽核軌跡

2. 玩家說分數或房卡對不起來，要回溯
- 客服要先查：玩家餘額、異動明細、來源類型、關聯訂單
- 客服要能做：限定權限的手動調整、強制備註
- 後台必要功能：房卡餘額、房卡 log、來源過濾、調整稽核

3. 玩家身分在公會內，要求建立/調整/解散公會
- 客服要先查：公會狀態、成員、主責窗口、歷史調整
- 客服要能做：發起建立、編輯參考設定、發起解散
- 主管要能做：審批建立、審批重大編輯、審批解散
- 後台必要功能：公會生命週期流程（建立/編輯/拆散）+ 成員管理 + 稽核

4. 玩家不屬於公會，要求加入或退出
- 客服要先查：玩家目前是否已有公會、是否有待審請求
- 客服要能做：加入/移出、標記窗口角色
- 後台必要功能：公會成員管理、衝突檢查、歷史紀錄

5. 玩家申訴帳號處置（停權/復權）
- 客服要先查：玩家近期行為、歷史處置
- 客服要能做：提出處置建議
- 主管要能做：實際執行停權/復權
- 後台必要功能：玩家狀態流轉、處置備註、稽核回放

## 二、市場常見營運服務（同類型社交/棋牌遊戲）
1. 玩家服務
- 玩家搜尋（ID/暱稱/電話/裝置）
- 今日新註冊、近 7 日留存、異常帳號
- 玩家狀態管理（正常/停權/解封）
- 玩家資產快照與異動明細（貨幣/道具）

2. 交易與核帳
- 儲值訂單查詢與審核
- 付款方式分流（人工/金流）
- 對帳差異追蹤（待確認、已補單）
- 可疑訂單標記與複審

3. 遊戲經濟與資產
- 主要資產餘額看板（總量/增減趨勢）
- 資產異動來源分析（儲值、消耗、補償、活動）
- 手動調整（需權限與稽核）

4. 社群/公會營運
- 建立公會申請與審批
- 公會停用/恢復
- 公會成員加入/移出
- 公會參考分數設定與歷程

5. 風控與稽核
- 全操作稽核軌跡（誰、何時、改了什麼）
- 高風險行為清單（大量調整、異常登入、短時間多次審核）
- 異常告警（規則觸發）

6. 客服支援
- 問題工單（玩家問題、處理狀態、結案）
- 常用處理模版（停權、補償、申訴）
- 玩家歷史操作整合視圖

7. 內容與活動運營
- 公告管理（上線/下線時間）
- 活動配置（參數、獎勵、開關）
- 分群投放（白名單/黑名單）

8. 監控與報表
- 每日核心 KPI（DAU、付費人數、ARPPU、轉化）
- 資料品質監控（欄位缺值、延遲）
- 系統健康（API、資料庫、排程）

## 四、建議後台資訊架構（玩家任務導向）

### 導覽分組
1. 客服今日待辦
- 今日新玩家
- 待核帳
- 待審公會請求
- 高風險事件

2. 玩家
- 玩家列表
- 今日加入
- 狀態管理
- 玩家資產快照

3. 交易
- 儲值訂單
- 審核工作台
- 對帳差異

4. 資產
- 房卡餘額
- 房卡異動
- 手動調整（有權限才顯示）

5. 公會
- 公會列表
- 建立申請
- 編輯申請
- 解散申請
- 待審批
- 成員管理

6. 稽核與風控
- 稽核查詢
- 高風險事件

7. 系統
- 後台帳號與權限
- 健康檢查
- Legacy 快照歷史

### 操作頁共通原則
1. 每頁固定四區塊
- 篩選列
- 統計條
- 主列表
- 操作/詳情抽屜

2. 每個按鈕要有三種狀態
- 可操作
- 權限不足（顯示原因）
- readonly 禁止（顯示原因）

3. 每個改動型操作都必帶
- 二次確認
- 必填備註
- 稽核追蹤編號回傳

## 五、從玩家需求反推的實作優先順序

### P0（先把營運真的用得起來）
1. 公會生命週期最小流程
- support 建立申請
- support 編輯申請（參考值、可見性）
- support 解散申請
- manager/owner 審批
- 核准後套用建立/編輯/解散
- 自動寫入 AuditLog

2. 玩家服務工作台
- 預設先看今日新玩家
- 玩家頁整合房卡餘額與最近異動摘要
- 支援一鍵匯出 CSV

3. 審核待辦首頁化
- Dashboard 顯示待審核帳單、公會建立/編輯/解散請求數

### P1（提高決策效率）
1. 公會成員管理（加入/移出/窗口標記）
2. 稽核頁新增進階篩選（操作者、實體、時間）
3. 高風險操作清單（短時間大量調整房卡、短時間多次公會異動）

### P2（治理與自動化）
1. 告警規則中心
2. 客服工單整合
3. KPI 報表中心

## 六、驗收標準（避免做了不好用）
1. 支援角色
- support、manager、owner 三層權限行為一致

2. 支援模式
- readonly 模式下所有寫入都要正確阻擋

3. 支援追溯
- 每個寫入行為都能在 AuditLog 反查

4. 支援日常效率
- 常見操作 3 次點擊內完成
- 今日重點任務 30 秒內看懂

## 七、玩家需求分流框架（每次新需求都先套）
每次先回答三件事：
1. 這是玩家哪一種事件（儲值、計分回溯、公會、身分狀態）
2. 客服需要哪個動作（查詢、提案、執行、回覆）
3. 哪個角色有最終執行權（support、manager、owner）

## 八、你現在可以直接採用的討論框架
每次提新需求時只回答四個欄位：
1. 這需求屬於哪一類（玩家/交易/資產/公會/風控/客服/系統）
2. 是查詢還是改動
3. 需要哪個角色可操作
4. 稽核要留哪些欄位

這樣討論不會再亂，且可直接轉為 API 與 UI 任務。

## 九、目前已落地的公會最小 API（2026-07-18）
1. 公會查詢
- GET /api/guilds：support / manager / owner
- GET /api/guilds/:id：support / manager / owner

2. 公會申請流程
- GET /api/guild-approvals：support / manager / owner
- POST /api/guild-approvals：support / manager / owner
- POST /api/guild-approvals/:id/approve：manager / owner
- POST /api/guild-approvals/:id/reject：manager / owner

3. 公會成員管理
- POST /api/guilds/:id/members：support / manager / owner
- PATCH /api/guilds/:id/members/:memberId/role：support / manager / owner
- POST /api/guilds/:id/members/:memberId/remove：support / manager / owner

4. requestType 與語意
- create_guild：建立公會（核准後才建立 Guild）
- update_reference_rate：更新公會參考設定（核准後才套用）
- revoke_guild：拆散/停用公會（核准後 guild.status = revoked）
